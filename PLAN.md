# PLAN.md

Roadmap for turning the freshly cloned `ns8-kickstart` repository into a production-capable NethServer 8 module that installs, configures, operates and upgrades the standalone Grantora Agent Capability Gateway.

## Goal

`ns8-grantora` packages upstream Grantora for NS8. It must not fork Grantora application logic.

Runtime target:

```text
Agent / automation client
  -> NS8 Traefik public route
  -> APISIX runtime endpoint in ns8-grantora
  -> grantora-api
  -> PostgreSQL dynamic state
  -> adapter -> upstream business application
```

NS8 module responsibilities:

- install and run upstream Grantora containers;
- generate static environment configuration from NS8 actions;
- store generated secrets in NS8 state;
- expose only APISIX runtime endpoints publicly through Traefik;
- keep Grantora Admin API, APISIX Admin API, PostgreSQL and etcd private;
- provide admin UI actions for setup, status, user-provider selection and common Grantora administration;
- bind to the selected NS8 user domain with `accountconsumer` authorization;
- sync selected LDAP users into Grantora users;
- back up and restore PostgreSQL plus environment-managed secret material;
- run APISIX reconciliation after configure, restore and upgrade;
- provide lifecycle, smoke, security and Robot tests.

Upstream Grantora responsibilities remain:

- runtime/admin API behavior;
- capability authorization;
- RBAC and bindings;
- adapter execution;
- secret encryption and resolution;
- audit and usage records;
- generated APISIX route desired state;
- filtered OpenAPI and MCP-compatible tool descriptions.

## Non-negotiable module rules

- Do not expose raw upstream application APIs.
- Do not expose Grantora Admin API publicly unless a future explicit private/operator route contract is added.
- Do not expose APISIX Admin API, PostgreSQL or etcd on public interfaces.
- Do not store upstream application secrets in agent configuration.
- Do not log tokens, passwords, authorization headers, encrypted secret values or decrypted secrets.
- Do not make upstream Grantora depend on NS8 paths, commands or Redis.
- PostgreSQL is Grantora source of truth; APISIX etcd is generated runtime state.
- Environment files are generated by the module; hand-edited config files are not a runtime contract.
- User provider selection is NS8-owned: the module lists available account domains, the admin selects one, and the module binds to it.

## Runtime exposure model

Use two loopback-only host ports if needed by NS8 actions and Traefik:

```text
127.0.0.1:<runtime_port> -> APISIX 9080       Traefik public route target
127.0.0.1:<admin_port>   -> grantora-api 8080 actions/UI-only operator API
```

Only `<runtime_port>` receives a Traefik route. `<admin_port>` is never exposed through Traefik by default.

Never publish these services to the host:

```text
PostgreSQL 5432
APISIX Admin API 9180
APISIX etcd 2379
```

If a later implementation avoids `<admin_port>` by executing admin calls inside the pod, keep the same public/private contract: only APISIX runtime endpoints are public.

## User provider integration design

- Add `org.nethserver.authorizations=cluster:accountconsumer` to the module image labels.
- Discover available account domains with `agent.ldapproxy.Ldapproxy().get_domains_list()`.
- In `get-configuration`, return the available domains with at least: domain name, schema, provider module when available, current selection flag, and whether the domain is reachable.
- In `configure-module`, require `user_domain` unless the module is explicitly configured in standalone/no-user-sync mode.
- Bind the selected domain with `agent.bind_user_domains([user_domain])`.
- Store the selected domain in `state/environment` as `LDAP_USER_DOMAIN` and in `state/user-domain.json` with sync metadata.
- Handle `user-domain-changed` and `module-domain-changed` events by scheduling or running `sync-users`.
- `sync-users` reads users through `agent.ldapclient` using filters from `agent.ldapproxy` so hidden users are not imported.
- Sync users into Grantora through Admin APIs using the LDAP username as Grantora `external_id`.
- Never pass LDAP passwords to Grantora and never store raw user passwords.

Initial sync behavior:

```text
LDAP user exists and Grantora user missing      -> create active Grantora user
LDAP user exists and Grantora user disabled     -> reactivate only if configured to do so
LDAP user disappeared/hidden/disabled           -> disable Grantora user if managed by selected domain
LDAP display name changed                       -> update when upstream supports metadata update
```

Until upstream bulk/idempotent sync exists, implement the action with paginated `GET /v1/admin/users`, `POST /v1/admin/users` and `PATCH /v1/admin/users/{id}` where possible.

## Upstream Grantora feature backlog required by ns8-grantora

Track these in `Stell0/grantora` before marking `ns8-grantora` production-ready.

### UP-1 — Versioned production database migrations

Reason: current pre-release schema bootstrap is acceptable for disposable development, not production upgrades.

Required upstream work:

- [ ] Add migration framework or equivalent versioned schema upgrader.
- [ ] Add CLI command or startup mode for migration execution.
- [ ] Document backup-before-upgrade and rollback boundaries.
- [ ] Add migration tests from previous released schema to current schema.
- [ ] Update `PROJECT.md`, `CONTRACTS.md`, `OPERATIONS.md` and `TESTING.md`.

### UP-2 — Idempotent/bulk user sync API

Reason: NS8 selected account domain can contain many users; module-side per-user create/list/patch loops are fragile and cannot update all metadata cleanly.

Required upstream work:

- [ ] Add `POST /v1/admin/users/sync` or equivalent idempotent API.
- [ ] Support source ownership, e.g. `source="ns8:<domain>"`.
- [ ] Support create/update/disable in one transaction per workspace.
- [ ] Return counts and safe errors only.
- [ ] Audit the sync summary.
- [ ] Add tests for workspace isolation and disabled user behavior.

### UP-3 — Mutable user metadata and optional group metadata

Reason: LDAP display names, mail addresses and future group-based binding helpers need metadata updates after initial creation.

Required upstream work:

- [ ] Allow safe update of `display_name` and optional `email`.
- [ ] Decide whether groups are stored as Grantora metadata or represented only by module-side binding helpers.
- [ ] Preserve runtime authorization deny-by-default.
- [ ] Add tests for metadata update without cross-workspace leakage.

### UP-4 — Stable release image contract

Reason: NS8 module releases must pin known-good upstream image tags and preferably digests.

Required upstream work:

- [ ] Publish release images under a stable registry namespace.
- [ ] Document supported tags and digest pinning policy.
- [ ] Keep `/healthz` version reporting.
- [ ] Keep release smoke test mandatory before publishing.

### UP-5 — Secret/key rotation support

Reason: module can generate initial secrets, but safe production rotation requires application support.

Required upstream work:

- [ ] Add encryption-key re-encryption workflow before exposing key rotation in NS8.
- [ ] Add token-pepper rotation workflow or document that pepper rotation invalidates all tokens.
- [ ] Add tests proving old encrypted secrets remain readable after re-encryption.

### UP-6 — Operator status API, only if module status needs more than current APIs

Reason: module UI should not parse logs to report operational status.

Current APIs already cover health, readiness, APISIX status, audit, usage and core object lists. Add more only if needed.

Potential upstream work:

- [ ] Add a concise `GET /v1/admin/status` returning safe aggregate status.
- [ ] Include dependency status, APISIX sync status, counts by object type and current version.
- [ ] Exclude secrets, tokens, raw upstream payloads and internal URLs.

## First implementation slice

Start with the smallest useful vertical slice:

1. Initialize repository naming and docs.
2. Package the four-container runtime.
3. Generate secrets/env files.
4. Configure public Traefik route to APISIX runtime only.
5. Start stack and sync APISIX.
6. Expose `get-status`.
7. Add UI provider selector backed by domain discovery.
8. Bind selected domain and run one-shot user sync.
9. Bootstrap a mock capability and invoke it through the public runtime route.
10. Add backup/restore smoke.

Do not start broad adapter UI work until this slice is green.
