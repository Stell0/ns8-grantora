# PLAN.md

Roadmap for turning the freshly cloned `ns8-kickstart` repository into a production-capable NethServer 8 module that installs, configures, operates and upgrades the standalone Grantora Agent Capability Gateway.

## Goal

`ns8-grantora` packages upstream Grantora for NS8. It must not fork Grantora application logic.

Runtime target:

```text
Agent / automation client
  -> NS8 Traefik public route
  -> 127.0.0.1:<runtime_port>
  -> Grantora Podman pod
  -> APISIX runtime port 9080
  -> grantora-api internal port 8080
  -> PostgreSQL internal port 5432
  -> adapter -> upstream business application
```

Upstream Grantora runtime components:

```text
grantora-api     FastAPI service: runtime API, admin API, invocation, audit, usage, APISIX reconciliation
postgres         source of truth for Grantora dynamic state
apisix           HTTP data-plane and runtime gateway
apisix-etcd      APISIX runtime configuration backend
```

NS8 module responsibilities:

- create one `grantora` Podman pod per module instance;
- publish only the APISIX runtime port from the pod to host loopback;
- launch each pod container with its own systemd user unit;
- generate static environment configuration from NS8 actions;
- store generated secrets in NS8 state;
- expose only APISIX runtime endpoints publicly through Traefik;
- keep Grantora Admin API, APISIX Admin API, PostgreSQL and etcd private inside the pod;
- provide admin UI actions for setup, status, user-provider selection and common Grantora administration;
- bind to the selected NS8 user domain with `accountconsumer` authorization;
- sync selected LDAP users into Grantora users;
- back up and restore PostgreSQL plus environment-managed secret material;
- run APISIX reconciliation after configure, restore and upgrade;
- provide lifecycle, smoke, security and Robot tests.

Upstream Grantora responsibilities remain:

- runtime/admin API behavior;
- capability authorization;
- RBAC and bindings;
- adapter execution;
- secret encryption and resolution;
- audit and usage records;
- generated APISIX route desired state;
- filtered OpenAPI and MCP-compatible tool descriptions.

## Non-negotiable module rules

- Do not expose raw upstream application APIs.
- Do not expose Grantora Admin API publicly unless a future explicit private/operator route contract is added.
- Do not expose APISIX Admin API, PostgreSQL or etcd on public interfaces.
- Do not publish individual container ports directly; port mapping belongs to `grantora-pod.service` only.
- Do not add a default host-mapped admin port. Admin actions must call Grantora from inside the pod.
- Do not store upstream application secrets in agent configuration.
- Do not log tokens, passwords, authorization headers, encrypted secret values or decrypted secrets.
- Do not make upstream Grantora depend on NS8 paths, commands or Redis.
- PostgreSQL is Grantora source of truth; APISIX etcd is generated runtime state.
- Environment files are generated by the module; hand-edited config files are not a runtime contract.
- User provider selection is NS8-owned: the module lists available account domains, the admin selects one, and the module binds to it.

## Pod and systemd topology

Grantora must run as one rootless Podman pod controlled by systemd user units.

```text
grantora.service                 aggregate service, starts/stops the whole module runtime
grantora-pod.service             creates/replaces the pod and owns all host port mappings
grantora-postgres.service        helper container in the pod
grantora-apisix-etcd.service     helper container in the pod
grantora-api.service             upstream Grantora API container in the pod
grantora-apisix.service          APISIX data-plane container in the pod
grantora-user-sync.timer         periodic LDAP -> Grantora user sync
grantora-user-sync.service       one-shot sync service
```

Systemd ordering:

```text
grantora.service
  Wants=grantora-pod.service
  Wants=grantora-postgres.service
  Wants=grantora-apisix-etcd.service
  Wants=grantora-api.service
  Wants=grantora-apisix.service

grantora-postgres.service
  Requires=grantora-pod.service
  After=grantora-pod.service

grantora-apisix-etcd.service
  Requires=grantora-pod.service
  After=grantora-pod.service

grantora-api.service
  Requires=grantora-pod.service grantora-postgres.service grantora-apisix-etcd.service
  After=grantora-pod.service grantora-postgres.service grantora-apisix-etcd.service

grantora-apisix.service
  Requires=grantora-pod.service grantora-api.service grantora-apisix-etcd.service
  After=grantora-pod.service grantora-api.service grantora-apisix-etcd.service
```

Pod rules:

```text
pod name: grantora
containers join with: --pod=grantora
published host ports: only 127.0.0.1:<runtime_port>:9080 by default
container-to-container URLs: use 127.0.0.1 inside the pod network namespace
```

Internal URLs inside the pod:

```text
DATABASE_URL=postgresql+psycopg://grantora:${POSTGRES_PASSWORD}@127.0.0.1:5432/grantora
APISIX_ADMIN_URL=http://127.0.0.1:9180
APISIX_RUNTIME_UPSTREAM_NODE=127.0.0.1:8080
APISIX_ETCD_ENDPOINT=http://127.0.0.1:2379
GRANTORA_API_INTERNAL_URL=http://127.0.0.1:8080
GRANTORA_RUNTIME_INTERNAL_URL=http://127.0.0.1:9080
```

Admin actions must not require a host admin port. Use one of these mechanisms:

```text
preferred: podman exec grantora-api python -m <safe module/admin helper>
accepted:  podman exec grantora-api python -c '<small urllib/httpx call to http://127.0.0.1:8080>'
accepted:  short-lived helper container attached to --pod=grantora when a stable helper image exists
rejected:  publishing grantora-api:8080 on the host by default
```

## Runtime exposure model

Default exposure:

```text
host loopback: 127.0.0.1:<TCP_PORT>
pod port:      APISIX 9080
route:         Traefik https://<host>/ -> http://127.0.0.1:<TCP_PORT>
```

Only `<TCP_PORT>` receives a Traefik route.

Never publish these services to the host:

```text
Grantora API 8080
PostgreSQL 5432
APISIX Admin API 9180
APISIX etcd 2379
```

Private/operator surfaces are reachable only from inside the pod through `grantora-admin`/`grantora-pod-exec` helpers:

```text
http://127.0.0.1:8080/v1/admin/*
http://127.0.0.1:8080/healthz
http://127.0.0.1:8080/readyz
http://127.0.0.1:8080/metrics when enabled
http://127.0.0.1:9180/* APISIX Admin API
```

If a future feature adds operator route exposure, it must be a separate explicit milestone with trusted-proxy/IP-allowlist/mTLS policy. It must not be added as an incidental configure option.

## User provider integration design

- Add `org.nethserver.authorizations=cluster:accountconsumer` to the module image labels.
- Discover available account domains with `agent.ldapproxy.Ldapproxy().get_domains_list()`.
- In `get-configuration`, return the available domains with at least: domain name, schema, provider module when available, current selection flag, and whether the domain is reachable.
- In `configure-module`, require `user_domain` unless the module is explicitly configured in standalone/no-user-sync mode.
- Bind the selected domain with `agent.bind_user_domains([user_domain])`.
- Store the selected domain in `state/environment` as `LDAP_USER_DOMAIN` and in `state/user-domain.json` with sync metadata.
- Handle `user-domain-changed` and `module-domain-changed` events by scheduling or running `sync-users`.
- `sync-users` reads users through `agent.ldapclient` using filters from `agent.ldapproxy` so hidden users are not imported.
- Sync users into Grantora through Admin APIs using the LDAP username as Grantora `external_id`.
- Never pass LDAP passwords to Grantora and never store raw user passwords.

Initial sync behavior:

```text
LDAP user exists and Grantora user missing      -> create active Grantora user
LDAP user exists and Grantora user disabled     -> reactivate only if configured to do so
LDAP user disappeared/hidden/disabled           -> disable Grantora user if managed by selected domain
LDAP display name changed                       -> update when upstream supports metadata update
```

Until upstream bulk/idempotent sync exists, implement the action with paginated `GET /v1/admin/users`, `POST /v1/admin/users` and `PATCH /v1/admin/users/{id}` where possible.
